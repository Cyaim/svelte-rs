# membench —— 帧成本 / 内存基准台

一个二进制、多场景。每个场景把**同一套渲染循环**跑在不同的树形与更新工况上,
最后打印一行可解析的统计并驻留(供外部采样进程内存)。

场景之间**刻意只差一个变量**:单个绝对数字没有意义(换台机器就变),
能相减的两个才有意义。所以下面每个场景都自带对照组,读数字先读差值。

```sh
cargo build --release -p membench          # 数字一律用 release 跑,dev 档是 opt-level=1
./target/release/membench --scene text --controls 2000 --mutate --frames 20 --hold-secs 0
```

## 通用参数

| 参数 | 缺省 | 说明 |
|---|---|---|
| `--scene rows\|virtual\|deep\|text\|scroll\|churn` | `rows` | 场景选择 |
| `--virtual` | — | 旧口径,等价于 `--scene virtual`(CI 与旧采样脚本在用,不会废) |
| `--controls N` | 3000 | 控件规模。行式场景按"每行 5 控件"折算行数;text 场景 = Text 节点数 |
| `--frames N` | 3 | 计时帧数(预热 1 帧不计入) |
| `--mutate` | 关 | 每帧驱动一次变更;不加 = 静止帧(走布局缓存) |
| `--backend cpu\|vello` | cpu | vello 需 `--features backend-vello`,离屏含纹理回读,略高估 |
| `--no-render` | 关 | 只建树不渲染(量建树时间/内存) |
| `--hold-secs N` | 6 | 打印后驻留,给外部采样 WorkingSet/Private |
| `--windowed` | 关 | 开真窗口(配 `SV_SHOW_FPS=1`),所有场景都支持 |
| `--depth N` | 200 | 仅 deep:嵌套层数 |
| `--text-pool N` | 0 | 仅 text:0 = 每帧生成没见过的串;N = 在 N 个串里循环 |
| `--wrap` | 关 | 仅 text:定宽容器 + 长段落折行 |
| `--unkeyed` | 关 | 仅 churn:换成不带 key 的 each(整表重建,当参照值) |

## 输出格式(**改前先看这条**)

```
READY backend=cpu scene=rows mutate=false virtual=false nodes=3001 signals=1201 \
      build_ms=0 warmup_ms=32 frame_avg_ms=14.15 p99_ms=16.29 low1_fps=61 fps=71 frames=20
```

CI 的帧预算闸(`.github/workflows/ci.yml` 的 `bench` job)用 `sed` 从这行抠
`p99_ms=`。**字段只增不改**——改名字、改分隔符、把 `READY` 挪出行首,都会让那道
闸静默变成永远通过。`scene=` 是本次新增的字段,加在既有字段之前,不影响解析。

## 本机实测(2026-07-22)

机器:i5-12400(12 线程)/ Windows 11 / release / CPU 后端 / 1920×1080 @1.0,
命令一律 `--frames 20 --hold-secs 0`,每档连跑 3–5 次取 `frame_avg_ms` 最小的
一次(**同一批次内连着跑完**,否则跨批次的机器状态差会盖过场景差)。

> **绝对值不要跨机器引用**。这台机器上跑基准时后台常有并行编译,同一条命令
> 前后能差 2 倍;但**同一批次内的对照关系**在所有批次里都稳定复现,所以下面
> 的结论都写成比值/差值,不写成"应当是 X ms"。
>
> **`p99_ms` 在这里其实是最大帧**:统计取 `sorted[(len*0.99) as usize]`,
> `--frames 20`(乃至 CI 的 30)下这个下标就是最后一个元素。字段名是历史包袱
> (改名会打断 CI 的 sed,见上一节),读的时候当"最坏帧"看,别当分位数看;
> 要真分位数得跑 `--frames 200` 以上。**比值一律用 `frame_avg_ms` 算**,
> 最大帧受调度抖动影响太大。

| 场景 | 命令(省略 `--frames 20 --hold-secs 0`) | nodes | avg ms | p99 ms | 1% low |
|---|---|---|---|---|---|
| 全量树(静止) | `--controls 3000` | 3001 | 14.34 | 22.29 | 45 |
| 全量树(每帧改一行) | `--controls 3000 --mutate` | 3001 | 20.25 | 24.25 | 41 |
| 虚拟化 100k(静止) | `--controls 100000 --virtual` | 152 | 3.18 | 3.91 | 255 |
| 虚拟化 100k(连续滚) | `--controls 100000 --virtual --mutate` | 152 | 3.66 | 4.30 | 232 |
| **深树 200 层** | `--scene deep --controls 3000 --depth 200 --mutate` | 3001 | 13.45 | 18.43 | 54 |
| 深树 4 层(对照) | `--scene deep --controls 3000 --depth 4 --mutate` | 3001 | 11.37 | 13.63 | 73 |
| **文本重排(全 miss)** | `--scene text --controls 2000 --mutate` | 2002 | 97.88 | 116.43 | 9 |
| 文本重排(池内,全 hit) | `--scene text --controls 2000 --text-pool 64 --mutate` | 2002 | 46.32 | 55.64 | 18 |
| 文本静止(对照) | `--scene text --controls 2000` | 2002 | 52.47 | 62.49 | 16 |
| 文本 500 不折行(对照) | `--scene text --controls 500 --mutate` | 502 | 23.77 | 26.37 | 38 |
| **文本 500 折行(全 miss)** | `--scene text --controls 500 --wrap --mutate` | 502 | 112.44 | 127.81 | 8 |

## 各场景:测什么、怎么读、什么回归会被它抓到

### `deep` —— 深树 / 父链回溯

**造什么**:`--depth` 层嵌套 View 串成一条链,节点总数仍由 `--controls` 定
(每层 = 1 个链节 + 若干叶子),所以**节点数不变、只有深度变**。字号与前景色
只写在根上,中间层一律不写 —— 继承解析必须走满整条链才收敛,这是最坏工况。
每层留一个绑定 Text 当探针,其余叶子是定尺色块。

**为什么叶子是色块不是文字**:绘制端 `text::shape` 没有缓存,一个文本叶子一帧
一次 parley 布局(几十 µs),几千个叶子的文本成本会把深度信号整个淹掉——
早期版本叶子用 Text 时,深度 200↔4 只差 20%,换成色块后差 40%。

**测的是** `sv-shell/src/render.rs` 里三条没有缓存的父链回溯:
`resolve_font_size` / `resolve_fg`(每个 Text 一趟)、`effective_opacity`
(**每个节点**一趟)。单帧代价是 O(节点数 × 深度)。

**怎么读**:只看 200 层 ÷ 4 层。本机 14.20 / 10.16 ≈ **1.40**——3000 个节点、
平均 100 层深,父链那部分约占单帧 4ms(28%)。

**会抓到什么**:
- 继承解析里多一次分配 / 多一趟 walk(比如"顺手"再解析一次背景色继承),
  比值立刻从 1.4 涨到 2+;
- 反过来,给 `effective_opacity` 加"整链都是 1 就早退"的快路,这里会**变快**
  —— 场景里 opacity≠1 每 64 层才放一个,就是为了不把这条优化路人为堵死;
- 缓存化父链解析(把继承值随布局一起下发)是最直接的优化,收益上限就是这个差值。

**踩到的坑**:本机 `--depth ≥ 400` 渲染时**爆栈**(`thread 'main' has overflowed
its stack`),`--depth 600 --no-render` 却安然无恙 —— 爆的是布局/绘制侧的递归
walker(`build_taffy` / `walk_taffy`),不是建树也不是析构。Windows 主线程默认
1MB 栈,~400 层就是天花板。缺省 200 留了 2 倍余量;真要压更深的树,得先把
walker 改成显式栈。

### `text` —— 文本重排 / Parley measure 缓存

**造什么**:`--controls` 个 Text 挂在一个容器下,所有绑定都读同一个帧号信号 →
一次写入改掉**全部**文本。

**三档对照**:
- `--text-pool 0`(默认):每帧生成从没见过的串 → measure 缓存全 miss,真跑 parley;
- `--text-pool 64`:在 64 个串里循环(远小于缓存容量 4096)→ 全 hit,量的是
  "改文本"本身的地板价(绑定重跑 + `set_text` + 版本 bump + 重绘);
- 不加 `--mutate`:静止帧,布局缓存整份命中。

**怎么读**(2000 个 Text):
- miss 91.68 − hit 46.86 = **44.8ms / 2000 = 每个文本每帧 ~22µs**,这就是
  measure 阶段 parley 布局的单价(taffy 两趟协议,每叶每帧问两次);
- **静止 52.14 ≈ 池内 46.86**:布局缓存全命中的静止帧,和每帧全量改文本的
  帧居然一个价 —— 因为地板是绘制端的 `text::shape`,它**没有缓存**,每帧每个
  文本节点都要重新 parley 一次。改文本本身几乎不要钱,前提是那个串见过。
- 折行档(500 个段落):不折行 27.14 → 折行 119.62,**4.4 倍**;折行池内
  49.67,即折行段落的 measure 单价 ≈ (119.62−49.67)/500 = **140µs/段/帧**,
  是短单行串的 6 倍多。

**会抓到什么**:
- measure 缓存被改坏(键里混进每帧都变的量、容量调小到装不下热集)→
  池内那档会塌到接近 miss 档;
- taffy 的测量协议从两趟变三趟、或折行宽在两趟之间抖动导致缓存键漂移 →
  wrap 档单价翻倍;
- 反过来,给绘制端 `shape` 加缓存(目前的头号优化机会)→ 静止档和池内档
  一起大幅下降,而 miss 档只降一半。这三档的相对关系就是验收标准。

### `scroll` —— 滚动搅动 / 布局缓存失效

**造什么**:一个真滚动容器(`overflow: Scroll` + 定高 900px 视口)里塞进
**全部** 600 行(刻意不虚拟化),每帧推一次像素偏移(取模回卷,免得撞到内容底
被布局钳住之后每帧画同一屏)。行内容与 `rows` 场景共用同一个行工厂,两者只差
"每帧改的是滚动位还是行数据"。

**测的是**:布局缓存是**按 Doc 版本号键控的整份产物**,滚动 = 一次 bump =
整棵树重新 measure/place + 重算裁剪矩形 + 重建滚动区元数据。

**怎么读**:滚动 21.75 vs 同规模改一行 19.18 —— **滚动帧和全量突变帧同价**
(+13% 是裁剪与滚动区簿记)。也就是说滚动这条路上**没有任何增量**:滚 1 像素
和改一行数据一样贵。对照同样 600 逻辑行的虚拟化档 3.48ms —— **6 倍**,而且
虚拟化档的节点数是 152 vs 3002。这就是 ADR-9 说的"帧成本与逻辑控件数解耦"。

**会抓到什么**:
- 有人给滚动加了"只平移不重布局"的快路 → 这里应该掉到接近静止帧,掉不下来
  说明快路没生效;
- 反之,布局缓存的键被改成"每帧必失效"(比如把时间戳混进 key)→ **静止档**
  会涨到滚动档的水平,这是最隐蔽的一类回归(功能全对,就是一直在重算);
- 滚动区簿记退化成 O(n²)(比如每帧为每个滚动容器重新扫全树算内容尺寸)→
  与 rows 的差值随规模放大。

### `churn` —— 结构搅动 / keyed each reconcile

**造什么**:`each_block_keyed` 列表,每帧整表**左旋 1**:key 一个不少,但顺序
全变 → reconcile 落到"逐行 append 对齐新序"的重排分支(append 是先从父的
children 里摘掉再推到末尾)。`--unkeyed` 换成不带 key 的 `each_block`:列表一变
就 `clear_children` + 整表重建 —— 它不是另一个工况,而是**复用一旦失效的参照值**。

**怎么读(这里的结论有点反直觉)**:600 行档 keyed 25.17 / unkeyed 22.80,
1200 行与 2400 行档也一样(46 vs 45、93 vs 88)—— **整表重建并不比整表重排贵**。
两层原因:① 建/毁场景节点很便宜(slotmap 插入 + 一次版本 bump);
② keyed 的重排自己也不便宜(按 key 线性查找 old 表是 O(n²),再加 n 次
`append` 的 O(n) 摘除),两边打平。

所以 keyed each 的价值是**状态保留**(行内 signal/effect 不重建、滚动位与焦点
不丢),不是帧成本;拿它当性能优化卖是错的。这条结论只有本场景能给出。

**会抓到什么**:
- 复用失效(改动让同 key 的行也走重建)→ keyed 档向 unkeyed 档靠拢**并越过**
  它,因为重建还会把行内绑定全部重跑一遍;
- reconcile 里出现真活儿(每行重新 diff 样式、每次重排触发 N 次布局失效)→
  keyed 档相对 `rows --mutate` 的差值(现在约 +6ms/600 行)按规模放大;
- key 查找从线性表换成哈希 → 大列表档应当明显变快,收益在这里可见。

## 方法学与坑

- **一律用 release**。dev 档是 `opt-level=1`(见根 `Cargo.toml`),文本栈与布局
  的常数差好几倍,拿 dev 数字讨论帧预算是浪费时间。
- **静止 ≠ 免费**。不加 `--mutate` 时布局缓存整份命中,但绘制每帧照做:
  `text::shape` 没缓存,所以文本多的界面静止帧也很贵(见 text 场景第三档)。
  这是目前最大的一块可回收成本。
- **规模天花板**:`--scene rows --mutate` 实测 3k → 19ms、10k → 58ms、
  24k → 349ms(节点数 2.4 倍、帧成本 6 倍,已经不是线性),与 DESIGN.md 里
  "30k 全量档 2ms 触发线已越"的结论一致 —— 上量只有虚拟化一条路。
- **CI 只卡两条线**(3k 全量 / 100k 虚拟,预算 120ms / 60ms,故意宽):CI 跑道
  是共享 vCPU,同一提交能差两三倍,硬卡本地数字只会制造 flaky。本文档的其余
  场景是**排查工具**,不是门槛;真要把某条加进 CI,先在 CI 上连跑几次看方差,
  再把预算定在中位数的 3–5 倍。
- **vello 后端**:所有场景都支持 `--backend vello`(需
  `--features backend-vello`),本轮没跑 —— 离屏 vello 含纹理回读,与 CPU 档
  不可直接比,要单独立基线。
