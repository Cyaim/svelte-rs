# membench —— 帧成本 / 内存基准台

一个二进制、多场景。每个场景把**同一套渲染循环**跑在不同的树形与更新工况上,
最后打印一行可解析的统计并驻留(供外部采样进程内存)。

场景之间**刻意只差一个变量**:单个绝对数字没有意义(换台机器就变),
能相减的两个才有意义。所以下面每个场景都自带对照组,读数字先读差值。

```sh
cargo build --release -p membench          # 数字一律用 release 跑,dev 档是 opt-level=1
./target/release/membench --scene text --controls 2000 --mutate --frames 20 --hold-secs 0
```

## 通用参数

| 参数 | 缺省 | 说明 |
|---|---|---|
| `--scene rows\|virtual\|deep\|text\|scroll\|churn` | `rows` | 场景选择 |
| `--virtual` | — | 旧口径,等价于 `--scene virtual`(CI 与旧采样脚本在用,不会废) |
| `--controls N` | 3000 | 控件规模。行式场景按"每行 5 控件"折算行数;text 场景 = Text 节点数 |
| `--frames N` | 3 | 计时帧数(预热 1 帧不计入) |
| `--mutate` | 关 | 每帧驱动一次变更;不加 = 静止帧(走布局缓存) |
| `--backend cpu\|vello` | cpu | vello 需 `--features backend-vello`,离屏含纹理回读,略高估 |
| `--no-render` | 关 | 只建树不渲染(量建树时间/内存) |
| `--hold-secs N` | 6 | 打印后驻留,给外部采样 WorkingSet/Private |
| `--windowed` | 关 | 开真窗口(配 `SV_SHOW_FPS=1`),所有场景都支持 |
| `--blit` | 关 | 仅 cpu 离屏:持久缓冲 + 脏矩形/scroll-blit(`render_into_cached`,事件循环同款),对照默认的全量 `render_frame` |
| `--depth N` | 200 | 仅 deep:嵌套层数 |
| `--text-pool N` | 0 | 仅 text:0 = 每帧生成没见过的串;N = 在 N 个串里循环 |
| `--wrap` | 关 | 仅 text:定宽容器 + 长段落折行 |
| `--unkeyed` | 关 | 仅 churn:换成不带 key 的 each(整表重建,当参照值) |

## 输出格式(**改前先看这条**)

```
READY backend=cpu scene=rows mutate=false virtual=false nodes=3001 signals=1201 \
      build_ms=0 warmup_ms=32 frame_avg_ms=13.18 p99_ms=16.11 low1_fps=62 fps=76 frames=20
```

CI 的帧预算闸(`.github/workflows/ci.yml` 的 `bench` job)用 `sed` 从这行抠
`p99_ms=`。**字段只增不改**——改名字、改分隔符、把 `READY` 挪出行首,都会让那道
闸静默变成永远通过。`scene=` 是本次新增的字段,插在 `backend=` 与 `mutate=`
之间,`^READY` 与 `p99_ms=` 两个锚点都没动,CI 的 sed 不受影响(两条 CI 命令
`--controls 3000` / `--controls 100000 --virtual` 都复跑过)。

其余单行输出(都**不带** `READY` 前缀,所以 CI 的 `grep '^READY'` 会因为抓不到
东西而失败退出,不会静默放行):

| 行 | 何时打 | 退出方式 |
|---|---|---|
| `SCENE-UNKNOWN <名>` | `--scene` 拼错 | 立即返回,不渲染 |
| `BACKEND-UNAVAILABLE <名>` | 后端名不认识,或 vello 探不到适配器 | 立即返回 |
| `DEPTH-WARN depth=<N> …` | `--scene deep --depth > 300` 且要渲染 | 只警告,继续跑(大概率随后爆栈) |

## 本机实测(2026-07-22)

机器:i5-12400(12 线程)/ Windows 11 / release / CPU 后端 / 1920×1080 @1.0,
命令一律 `--frames 20 --hold-secs 0`,每档连跑 3–5 次取 `frame_avg_ms` 最小的
一次(**同一批次内连着跑完**,否则跨批次的机器状态差会盖过场景差)。

> **绝对值不要跨机器引用**。这台机器上跑基准时后台常有并行编译,同一条命令
> 前后能差 2 倍;但**同一批次内的对照关系**在所有批次里都稳定复现,所以下面
> 的结论都写成比值/差值,不写成"应当是 X ms"。
>
> **`p99_ms` 在这里其实是最大帧**:统计取 `sorted[(len*0.99) as usize]`,
> `--frames 20`(乃至 CI 的 30)下这个下标就是最后一个元素。字段名是历史包袱
> (改名会打断 CI 的 sed,见上一节),读的时候当"最坏帧"看,别当分位数看;
> 要真分位数得跑 `--frames 200` 以上。**比值一律用 `frame_avg_ms` 算**,
> 最大帧受调度抖动影响太大。

| 场景 | 命令(省略 `--frames 20 --hold-secs 0`) | nodes | avg ms | p99 ms | 上一版 avg |
|---|---|---|---|---|---|
| 全量树(静止) | `--controls 3000` | 3001 | 13.24 | 15.33 | 13.18 |

| **文本重排(全 miss)** | `--scene text --controls 2000 --mutate` | 2002 | 88.21 | 97.38 | 93.71 |

> 3 次取 `frame_avg_ms` 最小的那次(见上文"取法")。"上一版 avg"是 `caea14c`
> 之后那一批,便于逐档比。

**怎么读这一版的变化:**

- ✅ **滚动档 20.32 → 12.45(1.63×)**,这是分级唯一预期会赢的一档:
  滚动偏移不进 taffy,现在复用布局树只重走产出坐标。
  单看布局那一段是 44 倍(29.05ms → 0.66ms,见 `docs/plans/incremental-layout.md`
  §11.5),但整帧只有 1.63 倍 —— **剩下的 12ms 全是绘制**。
  别拿子系统的比值当用户体感。
- 其余 `--mutate` 档基本持平或略好(rows 19.74 → 19.11、text 系全部下降):
  每帧都变更的负载走的是"整棵重建",跟以前同一条路,省下的只是原先
  每帧白克隆两份 `Layout` 的那点开销。
- ⚠️ **`deep` 两档与 `virtual 3000` 档比上一版慢约 3–10%,没有查明原因。**
  已排除"留着布局树导致的内存压力"(把 `KEEP_TREE_MIN_NODES` 设成
  `usize::MAX` 关掉留树,`deep` 一点没变)。也可能上一版基线本身偏乐观
  ——那批数字与本批之间隔了十来个提交,不只有分级这一件事。
  **写在这里而不是抹平,是因为"没查明"本身是这张表该记录的状态。**

> 顺带修掉一个真实开销:重建帧上新旧两棵布局树曾经同时活着(旧的要等
> `*slot = Some(..)` 才落地),峰值内存翻倍。改成先扔旧的再建新的,
> `rows 3k --mutate` 白拿 5%(20.0–21.0 → 19.0–19.3ms)。

## 各场景:测什么、怎么读、什么回归会被它抓到

### `deep` —— 深树 / 父链回溯

**造什么**:`--depth` 层嵌套 View 串成一条链,节点总数仍由 `--controls` 定
(每层 = 1 个链节 + 若干子节点),所以**节点数不变、只有深度变**。字号与前景色
只写在根上,中间层一律不写 —— 继承解析必须走满整条链才收敛,这是最坏工况。
绝大多数子节点是定尺色块,每 16 个插一个绑定 Text 当探针。

**为什么大多数是色块不是文字**:绘制端 `text::shape` 没有缓存,一个文本叶子
一帧一次 parley(短串实测 ~6µs),几千个文本叶子会把深度信号整个淹掉。

**为什么探针密度按节点数算、不是"每层一个"**(这条是本场景最容易搞错的地方):
每层放一个 Text 会让 Text 总数 = `--depth`,于是 200↔4 的对照里**除了深度还多
变了一个变量**——深档比浅档多 196 个 Text,那 196 次 shape 是纯加成、跟深度
一点关系没有。实测:每层一个 Text 时比值 13.6 : 9.9 = **1.38**;把 Text 数改成
与深度无关(每 16 个节点一个,两档都是 ~180 个)后 15.2 : 12.6 = **1.20**。
**虚报的那 0.18 全是文本成本**。同样的怀疑也核过 opacity(深档有 3 层
`opacity=0.92`、浅档一层都没有,alpha 混合会不会更慢?):去掉那几层后比值
1.38 → 1.38,纹丝不动,所以不必为它做等量化。

**测的是** `sv-shell/src/render.rs` 里三条没有缓存的父链回溯:
`resolve_font_size` / `resolve_fg`(每个 Text 一趟,`paint_tree` 里还各调了
不止一次)、`effective_opacity`(**每个节点**一趟)。单帧代价是 O(节点数 × 深度)。
量级上 `effective_opacity` 是大头 —— 它按节点数收费,Text 探针只有 ~6%。

**怎么读**:只看 200 层 ÷ 4 层,且**两档必须交错着连跑**(先跑 6 次深的再跑 6 次
浅的会被机器状态漂移污染)。本机交错 6 对:深档均值 13.06 / 浅档 10.83 ≈ **1.21**,
即 3000 个节点、平均 100 层深,父链那部分约占单帧 2.2ms(**17%**)。

**会抓到什么**:
- 继承解析里多一次分配 / 多一趟 walk(比如"顺手"再解析一次背景色继承),
  比值立刻从 1.2 涨到 1.5+;
- 反过来,给 `effective_opacity` 加"整链都是 1 就早退"的快路,这里会**变快**
  —— 场景里 opacity≠1 每 64 层才放一个,就是为了不把这条优化路人为堵死;
- 缓存化父链解析(把继承值随布局一起下发)是最直接的优化,收益上限就是这个差值。

**信噪比提醒**:1.21 只比这台机器的批内噪声(±5%,单次跑到过 1.08 也跑到过
1.61)高一档,所以判回归要看**交错多对后的均值**,单跑一次两档相减没有意义。
嫌信号弱可以 `--depth 300`(仍在爆栈线内)把 O(节点数×深度) 那项再抬 50%。

**踩到的坑**:本机 `--depth ≥ 400` 渲染时**爆栈**(`thread 'main' has overflowed
its stack`,退出码 127,**不打印任何上下文**),`--depth 600 --no-render` 却安然
无恙 —— 爆的是布局/绘制侧的递归 walker(`build_taffy` / `walk_taffy`),不是建树
也不是析构。Windows 主线程默认 1MB 栈,~400 层就是天花板(实测 300 正常、
400 必爆)。缺省 200 留了 2 倍余量;真要压更深的树,得先把 walker 改成显式栈。
`--depth > 300` 且要渲染时会先打一行 `DEPTH-WARN` 预告 —— 只警告不钳制,
因为哪天真有人把 walker 改成显式栈,得能跑到 400 层来验收。

### `text` —— 文本重排 / Parley measure 缓存

**造什么**:`--controls` 个 Text 挂在一个容器下,所有绑定都读同一个帧号信号 →
一次写入改掉**全部**文本。

**三档对照**:
- `--text-pool 0`(默认):每帧生成从没见过的串 → measure 缓存全 miss,真跑 parley;
- `--text-pool 64`:在 64 个串里循环(远小于缓存下限容量 4096 —— 容量现在是
  自适应的 [4096, 65536],见 `text.rs` 的 `next_cap`)→ 全 hit,量的是
  "改文本"本身的地板价(绑定重跑 + `set_text` + 版本 bump + 重绘);
- 不加 `--mutate`:静止帧,布局缓存整份命中。

**怎么读**(2000 个 Text):
- miss 93.71 − hit 43.60 = **50.1ms / 2000 = 每个文本每帧 ~25µs**,这就是
  measure 阶段 parley 布局的单价(taffy 两趟协议,每叶每帧问两次);
- **静止 48.07 ≈ 池内 43.60**:布局缓存全命中的静止帧,和每帧全量改文本的
  帧居然一个价 —— 因为地板是绘制端的 `text::shape`,它**没有缓存**(`shape()`
  直接调 `with_layout`,一行缓存都没有),每帧每个文本节点都要重新 parley 一次。
  改文本本身几乎不要钱,前提是那个串见过。
- 折行档(500 个段落):不折行 24.06 → 折行 106.02,**4.4 倍**;折行池内
  43.46,即折行段落的 measure 单价 ≈ (106.02−43.46)/500 = **125µs/段/帧**,
  是短单行串的 5 倍。

**会抓到什么**:
- measure 缓存被改坏(键里混进每帧都变的量、容量调小到装不下热集)→
  池内那档会塌到接近 miss 档;
- taffy 的测量协议从两趟变三趟、或折行宽在两趟之间抖动导致缓存键漂移 →
  wrap 档单价翻倍;
- 反过来,给绘制端 `shape` 加缓存(目前的头号优化机会)→ 静止档和池内档
  一起大幅下降,而 miss 档只降一半。这三档的相对关系就是验收标准。

### `scroll` —— 滚动搅动 / 布局缓存失效

**造什么**:一个真滚动容器(`overflow: Scroll` + 定高 900px 视口)里塞进
**全部** 600 行(刻意不虚拟化),每帧推一次像素偏移(取模回卷,免得撞到内容底
被布局钳住之后每帧画同一屏)。行内容与 `rows` 场景共用同一个行工厂,两者只差
"每帧改的是滚动位还是行数据"。

**测的是**:布局缓存是**按 Doc 版本号键控的整份产物**,滚动 = 一次 bump =
整棵树重新 measure/place + 重算裁剪矩形 + 重建滚动区元数据。

**怎么读**:滚动 20.32 vs 同规模改一行 19.74 —— **滚动帧和全量突变帧同价**
(+3% 是裁剪与滚动区簿记)。也就是说滚动这条路上**没有任何增量**:滚 1 像素
和改一行数据一样贵。对照同样 600 逻辑行的虚拟化档 3.53ms —— **5.8 倍**,而且
虚拟化档的节点数是 152 vs 3002。这就是 ADR-9 说的"帧成本与逻辑控件数解耦"。

**会抓到什么**:
- 有人给滚动加了"只平移不重布局"的快路 → 这里应该掉到接近静止帧,掉不下来
  说明快路没生效;
- 反之,布局缓存的键被改成"每帧必失效"(比如把时间戳混进 key)→ **静止档**
  会涨到滚动档的水平,这是最隐蔽的一类回归(功能全对,就是一直在重算);
- 滚动区簿记退化成 O(n²)(比如每帧为每个滚动容器重新扫全树算内容尺寸)→
  与 rows 的差值随规模放大。

**`--blit` 变体(scroll-blit / 脏矩形)**:同一场景改走事件循环同款的
持久缓冲 + 损伤渲染(`render_into_cached`),整数位移滚动帧只搬像素 +
重画露出的条。READY 行尾部追加 `blit=… blit_frames=… partial_frames=…
full_frames=… skip_frames=…`(字段只增不改;非 `--blit` 模式全为 0)。
实测(release,3000 控件、100 帧、13px 整数步,2026-07-23):

| 路径 | frame_avg_ms | p99_ms | 1% low |
|---|---|---|---|
| 默认(全量 `render_frame`) | 12.91 | 18.41 | 54fps |
| `--blit` | **2.18** | **3.35** | **299fps** |

**5.9 倍**;blit_frames=99/100(回卷那一帧位移超视口,按守卫降级整帧)。
开窗(`SV_SHOW_FPS=1 --windowed`,含 softbuffer 整窗转换/呈现):
`SV_DAMAGE=0` 55fps → 默认 **~100fps**——剩余天花板是 present 的整窗
逐像素转换,等 `present_with_damage`。**会抓到什么**:
blit 资格判定被改坏(隔离扫描漏判/整数判定松了)不会在这里现形——那归
sv-shell 的逐字节差分测试;这里抓的是**性能回归**:blit_frames 掉到个位数
说明入场券发不出去(量化没接上/快照失效太勤),avg 回到两位数说明
剔除或搬移本身退化。

### `churn` —— 结构搅动 / keyed each reconcile

**造什么**:`each_block_keyed` 列表,每帧整表**左旋 1**:key 一个不少,但顺序
全变 → reconcile 落到"逐行 append 对齐新序"的重排分支(append 是先从父的
children 里摘掉再推到末尾)。`--unkeyed` 换成不带 key 的 `each_block`:列表一变
就 `clear_children` + 整表重建 —— 它不是另一个工况,而是**复用一旦失效的参照值**。

**怎么读(这里的结论有点反直觉)**:600 行档 keyed 21.76 / unkeyed 21.40,
1200 行与 2400 行档也一样(42.1 vs 41.1、83.6 vs 81.3)——
**整表重建并不比整表重排贵**,甚至还便宜一丁点。

原因不是"两边都 O(n²)",而是**两边的 reconcile 都只占帧成本的一小块**:同规模
`rows --mutate` 基线 19.74ms,keyed 21.76(重排 +2.0ms)、unkeyed 21.40
(重建 +1.7ms),**90% 的帧时间是两档共有的布局 + 绘制**,reconcile 那点差别
被淹在里面。所以 keyed each 的价值是**状态保留**(行内 signal/effect 不重建、
滚动位与焦点不丢),不是帧成本;拿它当性能优化卖是错的。

**本场景压不到 O(n²) 的 key 查找**(容易看走眼,写清楚):
`each_block_keyed` 用 `old.iter().position(...)` 线性查 old 表——但左旋 1 之后,
每一次命中都落在**剩余 old 表的第 1 位**(取走一个,下一个又顶上来),单次
O(1)。实测 600→1200→2400 行是 21.2→42.1→83.6ms 的**线性**(每翻一倍恰好
翻一倍),两条理论 O(n²) 项(key 查找、`append` 的 `children.retain`)在这个
规模上都还没抬头。要压 key 查找得让顺序**逆序**或大步长旋转
(`rotate_left(n/2)`),本场景刻意没造那个工况——左旋 1 是 keyed 重排的
**下界**,先量下界。

**会抓到什么**:
- 复用失效(改动让同 key 的行也走重建)→ keyed 档向 unkeyed 档靠拢**并越过**
  它,因为重建还会把行内绑定全部重跑一遍;
- reconcile 里出现真活儿(每行重新 diff 样式、每次重排触发 N 次布局失效)→
  keyed 档相对 `rows --mutate` 的差值(现约 +2.0ms/600 行)按规模放大;
- **不会**抓到:key 查找换哈希。这条工况下线性查找本来就是 O(1),换哈希在
  这里量不出收益 —— 谁做了那个优化,得先给本场景加一个逆序档再来验收。

## 方法学与坑

- **一律用 release**。dev 档是 `opt-level=1`(见根 `Cargo.toml`),文本栈与布局
  的常数差好几倍,拿 dev 数字讨论帧预算是浪费时间。
- **静止 ≠ 免费**。不加 `--mutate` 时布局缓存整份命中,但绘制每帧照做:
  `text::shape` 没缓存,所以文本多的界面静止帧也很贵(见 text 场景第三档)。
  这是目前最大的一块可回收成本。
- **规模天花板**:`--scene rows --mutate` 实测 3k → 19.3ms、10k → 61.7ms、
  24k → 147ms(取各档最优;24k 档连跑四次 145–147,方差极小)。
  **这条是线性的**:每节点 6.4 / 6.2 / 6.1 µs,节点数 8 倍、帧成本 7.6 倍。
  天花板不是"某个规模上突然爆炸",而是**斜率本身太陡** —— 每节点 ~6µs,
  144Hz 的 7ms 预算只够 ~1000 个节点。与 DESIGN.md"30k 全量档已越触发线"
  的结论一致,上量只有虚拟化一条路;但**别把它说成超线性**,本机量不到
  (`measure` 缓存自适应之前 24k 档确实会因为缓存颠簸而超线性,那是
  commit `caea14c` 修掉的那个坑,不是布局本身的复杂度)。
- **CI 只卡两条线**(3k 全量 / 100k 虚拟,预算 120ms / 60ms,故意宽):CI 跑道
  是共享 vCPU,同一提交能差两三倍,硬卡本地数字只会制造 flaky。本文档的其余
  场景是**排查工具**,不是门槛;真要把某条加进 CI,先在 CI 上连跑几次看方差,
  再把预算定在中位数的 3–5 倍。
- **vello 后端**:所有场景都支持 `--backend vello`(需
  `--features backend-vello`),本轮没跑 —— 离屏 vello 含纹理回读,与 CPU 档
  不可直接比,要单独立基线。
- **改了 CLI 就得同步三处**:本文件、`src/main.rs` 的模块头注释,以及
  `docs/en/performance.md` + `docs/zh-CN/performance.md` 里的参数表与 READY
  字段表(那两份是镜像,改一边必须同步另一边)。**当前那两份还缺**
  `--scene` / `--depth` / `--text-pool` / `--wrap` / `--unkeyed` 与 `scene=`
  字段 —— 待补。
