## 2. 内存构成(归因拆解)

```
CPU @3000:199MB          vello @3000:788MB
├─ 字体(fontdue)~173MB   ├─ 字体(fontdue)          ~173MB
├─ 帧缓冲 pixmap    ~8MB   ├─ wgpu device+vello 管线  ~575MB ★本机 DX12 实测
├─ UI(0.5KB/控件) ~1.8MB  ├─ 回读缓冲+目标纹理       ~16MB
└─ 运行时基线      ~0.7MB  ├─ UI(同左)              ~1.8MB
                            └─ 场景编码等              余量
```

**三个阴性实验**(和阳性结论同等重要):
1. `SV_VELLO_AA=area`(去 MSAA 缓冲):内存 776→775MB,**MSAA16 不是内存元凶**;
   帧时长亦无差(7.5→8.3ms,噪声级)——AA 方式在此负载下两不敏感;
2. 帧数 1/10/60:774.7→775.5MB,**无逐帧泄漏**,~575MB 是 device+管线一次性固定成本;
3. (承 16 号)`SV_FONT_SUBST=0` 阴性:字体内存在轮廓急切解析,不在替换表。

启动侧:vello 预热帧 ~1.0s(管线编译;CPU 后端 ~0.7s 且全是字体解析)。

## 3. 分场景判决

### 轻量(≤3000 控件)
**CPU 后端压倒性胜出**:81fps@3000、增量内存 ≈2MB;vello 同档 42fps(离屏口径)
+ 575MB 固定成本,轻量场景完全不值。**实证 ADR-3b 分档正确**:轻量档就该是
CPU 系(未来 vello_cpu),GPU 后端连探测都不必。字体修复后轻量档总内存
可到 ~25MB。

### 复杂(3000+ 分量级)
- 当前离屏口径下 vello **未跑赢** CPU(42 vs 81 @3k;6.5 vs 8.8 @30k)。三重归因:
  回读串行化(≈7ms 地板+管线气泡)、每帧全量 scene 编码(vello 无脏区,
  调研 13 已预警)、本机 GPU 档位。**真实窗口路径(无回读)预期显著反超,
  需窗口内 FPS 计数器实测**——列为行动项,本报告不替 GPU 后端下"更慢"的结论;
- 两后端在 30k 档都不达 60fps → **虚拟列表 + 脏区/缓存才是复杂场景正解**,
  后端切换不是银弹(与调研 15"光栅是墙"结论互证);
- **100k 档新发现**:vello scene 超默认 device limits(192MB>128MB)——需在
  device 创建时定制 limits 或分块 scene,记录为 vello 后端工程项。

### 复杂 + 3D(按惯例:分析式 + 本机实测锚点)
16 号预算模型中"wgpu device 60–150MB(典型)"修正为**本机 DX12 实测 ~575MB**
(vello 全管线口径;纯用户 3D 管线会低一些,但同 device 共享时 UI 侧管线成本
已含在内——共享 device 恰是 `<surface3d>` 设计的默认形态)。方差警示:该值
随驱动/后端(DX12/Vulkan/Metal)波动大,跨机基准待 CI 矩阵。

## 4. 方法学注记与行动项

- vello 离屏 fps ≠ 窗口 fps:回读是基准工具的成本,不是产品路径的成本。
  行动:shell 加帧率计数(窗口标题/日志),补窗口口径数据;
- 行动:vello device limits 定制(100k 档)、场景编码增量化(RecordingPainter
  命令流 diff,调研 14 预留)、`SV_VELLO_AA` 保留为调优开关(area 在低端 GPU
  的差异待跨机验证);
- membench 用法归档:`--backend cpu|vello --controls N --frames M [--no-render]`,
  搭配 `SV_FONT_SUBST` / `SV_VELLO_AA` 做归因实验。
