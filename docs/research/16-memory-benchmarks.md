## 2. 内存归属拆解

```
总内存(3000 控件渲染态 177MB)
├─ 字体(fontdue 急切解析 msyh.ttc 18.8MB)……… ~173MB(97.7%)★
├─ 帧缓冲(1920×1080×4 pixmap)………………………… ~8MB
├─ 场景树+响应式图+绑定(3000 控件)……………… ~1.8MB
└─ 运行时本体(reactive/ui/shell 基线)…………… ~0.7MB
```

- **每控件净成本 ≈ 0.5KB**(三档不渲染回归:0.60 / 0.44 / 0.51 KB/控件,
  含 232B 场景节点摊分 + 0.4 信号/控件 + effect 闭包 + 文本串)。
  与结构体护栏测试(`ViewNode` 232B / `Style` 88B)吻合。
- **`SV_FONT_SUBST=0` 实验:阴性**(175.3 → 173.7MB,-1%)。替换表不是元凶,
  **急切字形轮廓解析才是**——止血无捷径,唯一真解是懒光栅文本栈
  (swash/Parley 或 Glifo),与调研 13/15 的 M1.5 提级判决一致。
- 字形覆盖度缓存(本轮新增)属 CPU 时间优化,内存增量按上限 ≈1.3MB,数据中不可见。

## 3. 分场景判决

### 轻量场景(≤3000 控件)
**UI 本体 2.5MB、构建 ≤1ms、二进制 1.3MB——除字体外全绿。**
字体占总内存 97.7%,是唯一不达标项。文本栈迁移后,该场景可承诺
**总内存 ≈ 15MB(4.4 基线 + 2.5 UI + ~8 帧缓冲)**量级。
渲染首帧 ~300ms 中 ~270ms 也是字体解析(启动时间同源同解)。

### 复杂场景(3000+,分量级 10k/30k/100k)
- **内存不是瓶颈**:线性 0.5KB/控件,10 万控件 UI 本体 52MB,信号图无超线性增长;
- **构建不是瓶颈**:10 万控件 32ms(编译期定点更新路线的红利——没有 VDOM/diff 堆);
- **瓶颈是 CPU 光栅**:全帧渲染时间 ~90ms/帧@3 万、~350ms/帧@10 万(扣除字体首载),
  实证调研 15 的判决:**复杂界面以 vello(GPU)为硬前置**,内存侧无须担忧;
  配套项(滚动+虚拟列表)可把"存在 10 万控件"降为"实例化数千",两头夹击。

### 复杂场景 + 3D(按惯例:分析式,不可实测)
合成通道尚不存在(调研 15;本轮已落地 `PainterCaps.external_texture` 预留位,
CPU 后端恒 false)。**预算模型**(供 M2.5 立项参照):
```
总内存 ≈ UI 基线(上表)+ wgpu Device/Queue(驱动相关,典型 60–150MB)
        + 交换链(W×H×4×2~3 buffering)+ 3D 侧资产(用户域)
        + surface3d 中转纹理(W×H×4 × 视口数)
```
可测前置:vello 后端(M2)+ `<surface3d>` ADR。届时基准台加
`--surface3d WxH` 档位即可复用本测试方法。

## 4. 行动结论

1. **文本栈迁移(M1.5)从"优先"升为"数据背书的第一优先"**:一项迁移同时回收
   97.7% 内存、~270ms 首帧、fontdue 停更风险;
2. 轻量/复杂场景的**内存承诺指标**写入 README 需等文本栈迁移后(否则 177MB 说不出口);
3. 复杂场景路线不变(vello + 滚动 + 虚拟列表),本报告提供了"内存无忧、光栅是墙"的量化依据;
4. membench 保留为常驻基准台(`--controls/--frames/--no-render/--hold-secs`),
   文本栈迁移后重跑本表对照,并接入 CI 冒烟(阈值:UI 本体 ≤0.7KB/控件)。
