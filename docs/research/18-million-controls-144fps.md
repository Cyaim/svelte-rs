# 18 · 百万控件 @ 144fps:诊断答卷、swash 迁移与虚拟化落地

> 2026-07-18。承接调研 17 的三个疑问(内存远超预期 / GPU 无收益 / 帧率普遍低)
> 与新目标"百万级控件 1% low 稳定 144fps"。本轮全部落地并实测:
> **fontdue → swash 文本栈迁移**(基线内存 198→27MB)、**`virtual_list` 视口虚拟化**
> (1M 逻辑控件 p99=5.28ms、1% low=174fps)、字形缓存分代淘汰、vello 100k 崩溃修复。
> 口径:membench(release,Windows 11 DX12,1920×1080 离屏)+ 窗口冒烟。

## 1. 调研 17 三个疑问的答卷

### Q1 内存为什么远超预期?—— 97% 是 fontdue 的急切解析,已消灭

fontdue 加载 msyh.ttc 时**急切解析全部 CJK 轮廓**(~173MB,调研 16 归因),这
不是 UI 架构的成本。迁移到 **swash 0.2.10**(skrifa/read-fonts 后端,零拷贝懒
解析)后:

| `--controls 0` 基线 | fontdue(前) | swash(后) |
|---|---|---|
| WorkingSet | 200.7MB | **27.4MB** |
| Private | 198.5MB | **20.5MB** |
| 首帧 warmup | 573ms | **11ms** |

UI 本身的边际成本没变:0.5KB/控件(10 万控件全量也只 +41MB)。
**轻量场景(≤3000 控件)整机 ≤30MB**,原"字体迁移前不承诺内存指标"的风险条目撤销。

### Q2 GPU 为什么没有收益?—— 三重口径污染,不是 GPU 没用

1. **离屏回读地板**:基准工具的 buffer 回读 ≈7ms 串行成本,产品路径没有;
2. **窗口 vsync 封顶**:vello surface 走 AutoVsync,轻负载被钉在 60fps
   (本轮窗口冒烟再次实证:vello 恒 60,CPU softbuffer 无 vsync 到 ~800);
3. **瓶颈在 CPU 侧共享路径**:每帧全量布局 + 全量场景编码,两后端一起付,
   30k 档谁的光栅都不是主项(调研 17 已证)。
   GPU 的真实价值在 CSS ⏳ 能力(渐变/模糊/阴影/transform)与大面积重绘,
   不在"每帧几百个小矩形"——这正是 ADR-3b 分档策略的预设。

### Q3 帧率为什么普遍低?—— 全量重算,答案是"别全量"(见 §3)

顺带:swash 迁移后 CPU 光栅路径本身也快了约一倍(30k 档 113→47ms;字形
alpha 覆盖度直绘 + ScaleContext 复用),但这只是常数改善。百万级别的答案
只能是**架构性削减每帧工作集**。

## 3. 百万控件 @144fps:视口虚拟化落地与验收

### 设计(`sv_ui::virtual_list`)

144fps ⇒ 帧预算 6.94ms,p99/1% low 达标意味着**帧成本必须与逻辑控件数解耦**。
浏览器/移动端的标准答案是视口虚拟化,本轮落为 sv-ui 原语:

- **固定槽位**:视口 N 行只建 N 份真实节点,每槽持一个 `Signal<Option<T>>`;
- **滚动 = 数据置换**:offset 变化只做逐槽 `.set()`,经 bind_text 定点更新,
  **零节点创建/销毁、零结构变化**——这正是 1% low 稳定性的来源
  (没有"滚动帧偶发重建"的长尾);
- **懒取数**:`item_at(i)` 按需取,1M 条目永不物化;
- 单测:1M 逻辑行 → 场景树 ≤34 节点,`offset.set(500_000)` 原地更新。

### 验收(每行 5 控件 × 20 万行 = 100 万控件;视口 30 行;`--mutate` 每帧滚动一行 = 最坏工况)

| 口径 | 帧均 | p99 | **1% low** | WS |
|---|---|---|---|---|
| CPU 离屏 1920×1080,500 帧 | 3.41ms | **5.28ms** | **174fps** | 28.2MB |
| CPU 窗口(softbuffer 无 vsync) | — | — | **~800fps**(FPS 计数) | 37.9MB |
| vello 离屏(回读地板 ~7ms) | 9.26ms | 16.3ms | 56fps | 633MB |
| vello 窗口 | — | — | 60fps(vsync 钉死) | 598MB |

**判决:目标达成**——p99 5.28ms < 6.94ms 预算,1% low 174fps > 144fps,且内存
28MB(逻辑控件数不影响内存)。vello 窗口口径受 vsync 上限,需 mailbox/immediate
present mode 才能展示 >60(工程项,非能力问题;离屏扣除 7ms 回读后 ≈2.3ms/帧同样达标)。

### 尖峰治理:字形缓存分代淘汰

原字形缓存超限(2048 条)**整体清空**,清空后那一帧要全量重光栅——正是伤
1% low 的长尾来源。改为**两代(hot/cold)淘汰**:热代满则整代降冷、旧冷代丢弃,
活跃字形要么在热代、要么命中时从冷代零成本晋升;单帧最多重光栅"整代未用"的
字形,工作集永不被打掉。内存上界 2×CAP ≈ 2.6MB@16px。

## 4. 顺手修复:vello 100k 档崩溃(device limits)

vello `RenderContext` 固定 `Limits::default()`(存储缓冲绑定上限 128MB),100k
控件的 scene buffer ≈192MB 直接触发 validation panic(调研 17 遗留)。离屏路径
没有 surface 兼容性约束 → **自建 device**,把 `max_storage_buffer_binding_size`
/`max_buffer_size` 抬到 adapter 实际能力。实测 100k 不再崩溃(452ms/帧——慢而
正确;该档位的正解本来就是 §3 的虚拟化,不是全量建树)。窗口路径仍走
RenderContext(受同一默认上限),记录为上游工程项。

## 5. 结论与下一步

1. **三大疑问全部闭环**:内存=fontdue(已迁移)、GPU 无收益=口径+分档预期
   (已解释并再实证)、低帧率=全量重算(虚拟化落地后 1M 达标);
2. **144fps 的架构公式**:虚拟化(每帧工作集 = 视口)+ 定点更新(零结构变化)
   + 缓存无尖峰(分代淘汰)。当前 CPU 后端即达标;
3. 后续(超出本轮验收,按优先级):
   - 帧调度 ADR-6(vsync 对齐 + mailbox 选项,让窗口口径能展示 >60);
   - 增量场景编码(RecordingPainter 命令流 diff,让**全量档**也降帧成本);
   - 局部布局(dirty 子树 relayout,当前靠版本键全量缓存,滚动帧仍全量布局
     ——虚拟化下节点少所以无感,复杂静态界面会感);
   - 滚动物理(惯性/像素级 offset)接到 virtual_list 之上。
